# AI Captain 飞书多维表格方案 V2

## 1. 核心变化

V1 的问题是字段齐，但还是偏“点状管理”。

看得到：

- 当前在做
- 当前卡点
- 最新反馈
- DDL

但还看不清：

- 项目正沿着哪条主线往前走
- 现在卡在线的哪一段
- 离下一个成熟阶段还差什么
- 多个项目放在一起时，资源该投给谁

所以 V2 改成两套骨架：

- `点线面`
- `滑板车 -> 自行车 -> 汽车 -> 高铁`

---

## 2. 点线面定义

### 点

单个状态点，回答“现在发生了什么”。

例如：

- 当前在做
- 当前卡点
- 最新反馈
- 下个检查点
- DDL
- Eval 分

### 线

项目演进路径，回答“这个项目正往哪走，卡在哪一段”。

V2 统一用成熟度主线：

`滑板车 -> 自行车 -> 汽车 -> 高铁`

### 面

项目组合视图，回答“这批项目整体分布如何，资源该怎么投”。

例如：

- 哪些项目还在滑板车
- 哪些项目已经到汽车
- 哪些项目业务价值高但成熟度低
- 哪些项目成熟度高但采用度低

---

## 3. 载具模型定义

### 滑板车

定义：

- 最小可跑通
- 能证明方向
- 依赖人工强
- 主要用于演示和验证

判断标准：

- 有 1 个清晰场景可以跑通
- 有 captain
- 有基本反馈来源
- 没有完整 SOP / Eval / 运维

### 自行车

定义：

- 能连续跑一个流程
- 开始有稳定使用场景
- 但还不足以正式交付

判断标准：

- 流程可重复运行
- 有周更新
- 有基础 Eval
- 有明确卡点
- 仍缺交付级监控、回滚或质量门槛

### 汽车

定义：

- 可以正式交付
- 可稳定承载一个团队或一条业务线

判断标准：

- Eval 达上线线
- 有 SOP
- 有监控和回滚
- 有值班 owner
- 有业务结果

### 高铁

定义：

- 可规模化复制
- 多团队接入后仍稳定
- 已经不是“一个项目”，而是“可复制能力”

判断标准：

- 多团队使用
- 复制成本低
- 质量和成本稳定
- 运维标准化
- 模板化扩张

---

## 4. 飞书 Base 结构

仍然建议保留 6 张表，但主逻辑改成“升级管理”：

1. `项目主表`
2. `升级条件表`
3. `任务表`
4. `周更新表`
5. `Eval 表`
6. `运维事件表`

重点变化是：新增 `升级条件表`，把“线”显式化。

---

## 5. 表 1：项目主表

每个项目一行，但现在不只是记录状态，而是记录“在哪个载具阶段、为什么卡住、离升级差什么”。

### 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| 项目ID | 文本 | 唯一键 |
| 项目名称 | 文本 | 展示名 |
| Captain | 人员 | 第一责任人 |
| Sponsor | 人员 | 业务发起人 |
| 所属业务线 | 单选 | 招聘 / 销售 / 内容 / 其他 |
| 当前载具阶段 | 单选 | 滑板车 / 自行车 / 汽车 / 高铁 |
| 目标载具阶段 | 单选 | 下一目标阶段 |
| 当前阶段状态 | 单选 | 绿灯 / 黄灯 / 红灯 |
| 目标DDL | 日期 | 交付目标日期 |
| 距DDL天数 | 公式 | 日期差 |
| 当前主线 | 长文本 | 当前阶段主任务是什么 |
| 当前在做 | 长文本 | 这周/当前正在推进什么 |
| 当前卡点 | 长文本 | 当前最大阻塞 |
| 最新反馈 | 长文本 | 最近反馈 |
| 反馈来源 | 文本/人员 | 谁提的 |
| 下个检查点 | 长文本 | 下次验证点 |
| 升级差距数 | 数字/公式 | 离下一阶段还差几项 |
| 是否可升级 | 单选 | 可升级 / 不可升级 / 待评估 |
| 总进度说明 | 长文本 | 用文字解释当前进度，而不是只给百分比 |
| GitHub 仓库 | URL | repo |
| GitHub Project | URL | project |
| Notion 文档 | URL | 文档 |
| WAU | 数字 | 周活跃用户 |
| 周任务量 | 数字 | 周有效任务数 |
| 节省人小时 | 数字 | 周节省时长 |
| 交付分 | 数字 | 0-100 |
| 质量分 | 数字 | 0-100 |
| 运维分 | 数字 | 0-100 |
| 采用分 | 数字 | 0-100 |
| 最后更新时间 | 日期时间 | 最近更新时间 |

### 这张表要直接回答

- 这个项目现在是滑板车、自行车、汽车还是高铁
- 它现在正沿哪条主线推进
- 离下一阶段还差几项
- 当前卡在哪
- 最近谁给了什么反馈

---

## 6. 表 2：升级条件表

这是 V2 最关键的新表。

它专门把“线”拆出来。

### 作用

不再模糊说“这个项目 68% 了”，而是明确说：

- 当前是 `自行车`
- 目标是 `汽车`
- 升级要满足 5 个条件
- 现在完成了 2 个
- 卡在 `Eval` 和 `回滚预案`

### 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| 条件ID | 文本 | 唯一键 |
| 关联项目 | 关联记录 | 关联项目主表 |
| 当前载具阶段 | 单选 | 当前阶段 |
| 目标载具阶段 | 单选 | 目标阶段 |
| 升级条件名称 | 文本 | 例：候选人识别达标 |
| 条件分类 | 单选 | 交付 / 质量 / 运维 / 采用 / 成本 |
| 当前状态 | 单选 | 未开始 / 进行中 / 已完成 / 阻塞 |
| Owner | 人员 | 负责人 |
| 验收标准 | 长文本 | 什么叫完成 |
| 当前问题 | 长文本 | 为什么还没过 |
| 预计完成时间 | 日期 | 预计完成时间 |
| 升级证据 | 长文本 / URL | 文档、数据、截图、复盘 |

### 示例

AI 招聘助手，自行车 -> 汽车，升级条件可能是：

1. 候选人识别准确率达标
2. 回滚预案确认
3. 值班负责人明确
4. 业务 SOP 完成
5. 周报摘要自动化上线

### 推荐视图

- `按项目看升级条件`
- `阻塞中的升级条件`
- `本周应完成升级条件`
- `按载具阶段分组`

### 字段设计

| 字段名 | 类型 | 说明 |

| 任务ID | 文本 | 唯一键 |
| 关联项目 | 关联记录 | 项目主表 |

## 11. 视图设计

这部分就是“面”。

### 视图 1：载具分层视图

按 `滑板车 / 自行车 / 汽车 / 高铁` 分组。

你会立刻看到：

- 哪些项目还只是验证阶段
- 哪些项目已可交付
- 哪些项目已经适合扩张

### 视图 2：升级路径视图

按项目展开升级条件。

能看到：

- 每个项目离下一阶段差几项
- 卡在哪种类型的条件上
- 是质量问题、运维问题，还是反馈/采用问题

### 视图 3：风险视图

过滤：

- 当前状态 = 红灯
- 升级条件 = 阻塞
- 是否需要升级介入 = 是

### 视图 4：组合决策面

建议在飞书里用分组 + 数字字段近似实现：

- 横向理解：成熟度（滑板车 -> 高铁）
- 纵向理解：业务价值（低 -> 高）

这张视图回答：

- 哪些值得继续加资源
- 哪些应该先停
- 哪些可复制

## 12. 推荐的项目卡呈现方式

以后看项目，不再先看百分比，而先看：

### 卡片结构

- 项目名称
- 当前载具阶段
- 目标载具阶段
- 当前主线
- 当前卡点
- 最新反馈
- 升级差距数
- 下个检查点

### 示例

AI 招聘助手

- 当前：`自行车`
- 目标：`汽车`
- 当前主线：补齐候选人识别标注，重跑 Eval
- 当前卡点：候选人识别 0 分；回滚预案未锁定
- 最新反馈：妃姐要求 captain / eval / deadline / 周进展清晰化
- 升级差距：`3 项`
- 下个检查点：周五前完成候选人识别复测

这就比“68%”直观得多。

## 13. 自动化规则

### 自动化 1：GitHub 同步

同步到 `任务表`，再关联到 `升级条件表`。

### 自动化 2：升级差距自动计算

统计某项目下：

- `未完成` 或 `阻塞` 的升级条件数

回填到 `项目主表.升级差距数`

### 自动化 3：周更新摘要

每周自动汇总：

- 当前在做
- 当前卡点
- 最新反馈
- 本周完成了几项升级条件
- 下个检查点

### 自动化 4：升级提醒

触发条件：

- 升级差距数 <= 1
- 或关键条件阻塞超过 7 天

提醒对象：

- Captain
- Sponsor

---

## 14. 最终判断标准

V2 不再问“项目多少分”这么抽象的问题。

而是问：

- 这个项目现在是滑板车、自行车、汽车还是高铁
- 离下一级还差哪些条件
- 卡点在哪
- 最近反馈是什么
- 值不值得继续投资源

这才是 `点线面` 的落法。

---

## 15. 实施顺序

1. 改造 `项目主表`，加入载具字段和升级字段
2. 新建 `升级条件表`
3. 改造 `任务表`，挂到升级条件
4. 改造 `周更新表`
5. 让 Eval 和运维事件能标记“是否阻塞升级”

如果只能做一步，先做：

`项目主表 + 升级条件表`

因为这一步最能把“线”拉出来。
