# AI Captain 飞书多维表格方案 V1

## 1. 结论

第一阶段不单独做 React 主看板。

正式执行面以飞书多维表格为主，原因很直接：

- 不增加额外部署和运维面
- 群里协作、提醒、责任人推进天然顺手
- 非技术成员可直接更新状态、反馈、阻塞
- 适合承载 captain / deadline / blocker / feedback / eval 这类管理信息

Notion 继续做文档层，不做执行主系统。

## 2. 飞书 Base 结构

建议建一个 Base：

- `AI Captain 项目总台`

V1 建 6 张表，够用且不散：

1. `项目主表`
2. `任务表`
3. `里程碑表`
4. `Eval 表`
5. `周更新表`
6. `运维事件表`

---

## 3. 表 1：项目主表

这是总入口，每个项目一行。

### 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| 项目ID | 文本 | 唯一键，格式 `captain-xxx` |
| 项目名称 | 文本 | 项目展示名 |
| Captain | 人员 | 第一责任人 |
| Sponsor | 人员 | 业务发起人/决策人 |
| 所属业务线 | 单选 | 招聘 / 销售 / 内容 / 其他 |
| 当前阶段 | 单选 | 立项 / 研发 / 交付 / 运维 / 扩张 / 暂停 |
| 项目状态 | 单选 | 绿灯 / 黄灯 / 红灯 |
| 目标DDL | 日期 | 目标交付日期 |
| 距DDL天数 | 公式 | `目标DDL - 今天` |
| 当前在做 | 长文本 | 现在正在推进什么 |
| 当前卡点 | 长文本 | 当前最关键阻塞 |
| 最新反馈 | 长文本 | 最近一次反馈原文或摘要 |
| 反馈来源 | 文本/人员 | 谁给的反馈 |
| 下个检查点 | 长文本 | 下次要验证什么 |
| 本周推进 | 数字 | 本周进度变化，例 `+8` |
| 总进度 | 数字 | 0-100 |
| 风险级别 | 单选 | 低 / 中 / 高 |
| 项目说明 | 长文本 | 一句话目标 |
| GitHub 仓库 | URL | 对应 repo |
| GitHub Project | URL | 对应 project board |
| Notion 文档 | URL | PRD / 方案 / 复盘 |
| WAU | 数字 | 周活跃使用人数 |
| 周任务量 | 数字 | 周有效任务数 |
| 节省人小时 | 数字 | 周节省的人小时 |
| 质量分 | 数字 | 0-100 |
| 运维分 | 数字 | 0-100 |
| 采用分 | 数字 | 0-100 |
| 交付分 | 数字 | 0-100 |
| 最后更新时间 | 日期时间 | 最近同步/人工更新时间 |

### 这张表要回答的问题

- 项目现在在哪个阶段
- 谁负责
- 当前到底在做什么
- 卡点是什么
- 最近谁提了什么反馈
- 什么时候要下一次检查

### 推荐视图

- `全部项目`
- `按 Captain`
- `红黄灯项目`
- `临近DDL`
- `交付中项目`
- `运维中项目`

| GitHub 链接 | URL | issue / pr |
| 创建时间 | 日期时间 | 创建时间 |
| 更新时间 | 日期时间 | 更新时间 |

### 推荐视图

- `关键路径任务`

- `阻塞任务`

## 9. 关键公式建议

### 距DDL天数

如果飞书公式支持日期差，直接用日期差。

语义规则：

- `< 0`：延期
- `0 - 14`：临近DDL
- `> 14`：正常

### 项目状态

建议先人工维护，不要一开始全自动推导。

规则参考：

- `红灯`
  - 延期且无明确恢复计划
  - 有关键路径阻塞
  - Eval 关键项低于上线阈值
- `黄灯`
  - 有风险但本周有推进
  - 交付前关键项未完全闭环
- `绿灯`
  - 无关键阻塞
  - 下个检查点明确
  - 推进正常

### 总进度

V1 不建议直接按任务完成数算。

建议按阶段 + 里程碑结合：

- 立项：10
- 研发：35
- 交付：70
- 运维：85
- 扩张：100

再叠加里程碑完成比例做微调。

## 10. 自动化流程

### 自动化 1：GitHub 任务同步到任务表

来源：

- GitHub Project
- Issues
- PR

写入：

- `任务表`

同步字段：

- 标题
- 状态
- assignee
- 链接
- 更新时间

### 自动化 2：每周生成项目摘要

来源：

- 项目主表
- 周更新表
- Eval 表
- 运维事件表

输出：

- 飞书群消息 / 卡片

内容：

- 当前在做
- 当前卡点
- 最新反馈
- 本周推进
- 下个检查点

### 自动化 3：DDL 和阻塞提醒

触发条件：

- 距DDL <= 7 天
- 项目状态 = 红灯
- 存在关键路径阻塞

输出：

- 飞书提醒 Captain / Sponsor

### 自动化 4：运维异常提醒

触发条件：

- 新增 P1/P2 运维事件

输出：

- 飞书项目群 / 负责人提醒

## 11. 为什么先不用 React

因为当前最重要的问题不是“缺一个前端”，而是：

- 信息该由谁维护
- 信息放在哪张表
- 哪些字段自动同步
- 哪些字段必须人工更新
- 哪些条件触发提醒

这些问题飞书多维表格天然更适合先解决。

等 V1 跑通以后，如果你们需要：

- 跨表图表
- 历史趋势
- 成本与质量聚合
- GitHub + 飞书 + 日志三方统一视图

再考虑加独立看板。
